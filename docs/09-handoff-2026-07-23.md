## 4. 当前上传协议

浏览器上传一个文件时的完整调用链：

```text
用户选择文件
→ 浏览器取得 File 对象
→ POST /api/v1/uploads
→ 服务端创建或恢复上传事务
→ 返回 upload_id、offset、chunk_size
→ 浏览器用 File.slice() 选取一个范围
→ arrayBuffer() 把该范围读入连续内存
→ PATCH /api/v1/uploads/{upload_id}?offset=...
→ 服务端接收完整请求体
→ 写入临时文件并返回新 offset
→ 浏览器串行发送下一分块
→ POST /api/v1/uploads/{upload_id}/complete
→ 服务端 fsync
→ os.replace 原子发布正式文件
→ 返回完整文件 SHA-256
```

需要特别注意：

- 选择文件时，Edge 通常不会把整个文件一次性装入内存；`File` 是浏览器提供的受控文件对象。
- `file.slice(start, end)` 是 Web API，描述一个文件范围，通常不会立刻读取全部字节。
- `arrayBuffer()` 才会把该范围读取到 JavaScript 可访问的连续内存。
- 当前默认分块是 16 MiB。
- 当前代码在发送本块时预读下一块，因此 JavaScript 层可能同时持有约两个分块，即约 32 MiB，此外还有浏览器内部网络缓冲。
- 当前同一文件只有一个 `PATCH` 请求正在传输，下一块只被预读，没有并行发送。

## 5. 上传会话、断点和分块的含义

上传会话是 CHFS 的业务状态，不等于一条持续存在的 HTTP 或 TCP 连接。

服务端根据 `upload_id` 保存类似以下状态：

```text
目标文件路径
临时文件路径
声明的文件总大小
已经写入的 offset
完整文件 SHA-256 状态
上传所有者
最后活动时间
```

- `offset` 表示服务端已经确认写入的字节数，用于断点续传、顺序校验和防止重复写入。
- `chunk_size` 表示客户端每次请求应发送多少数据。
- 分块太小会增加 HTTP 请求数量和确认开销。
- 分块太大会增加内存占用以及失败后的重传成本。
- 16 MiB 是合理基线，但不是经过当前设备实测得到的最优值。
- 服务端目前允许的单块上限是 32 MiB。

## 6. 已识别的性能瓶颈

尚未实施性能优化，当前只完成分析和注释。

### 浏览器端

1. 分块请求严格串行：

```javascript
session = await sendChunkWithRetry(...)
```

当前请求必须完整发送并取得服务端响应后，才会发送下一块。

2. 每个分块都通过：

```javascript
file.slice(position, end).arrayBuffer()
```

读入 JavaScript 连续内存，可能产生读取和复制成本。

3. 预读只隐藏部分本地文件读取等待，不能形成多个网络请求的传输流水线。

### 服务端

`ResumableUploadManager.append(...)` 当前会：

1. 把整个 HTTP 分块请求拼接进 `bytearray`。
2. 可选地同步计算分块 SHA-256。
3. 在全局 `RLock` 内同步写入临时文件。
4. 同步更新完整文件 SHA-256。

这些同步 CPU 和磁盘操作运行在异步请求处理路径中，可能阻塞事件循环线程。

最终提交时：

```python
os.fsync(stream.fileno())
```

会等待文件数据刷新到持久化设备，也属于同步等待。

## 7. 当前网络测试事实

在本机检查到的当前无线链路：

- 电脑无线网卡：MediaTek Wi-Fi 6E MT7902。
- 当前连接协议：`802.11ax`，即 Wi-Fi 6，而不是 Wi-Fi 7。
- 频段：5 GHz。
- 协商发送/接收速率：约 600.5 Mbps。
- PHY 换算上限约 75 MB/s，不等于文件传输实际速度。
- 手机上传实测约 17～25 MB/s，即约 136～200 Mbps。

因此软件存在瓶颈嫌疑，但尚不能只凭文件上传速度排除手机热点、无线链路和浏览器实现的限制。

建议先用 `iperf3` 或同类工具测试相同设备和链路的纯 TCP 吞吐：

- 若纯 TCP 明显高于 25 MB/s，重点优化 CHFS。
- 若纯 TCP 也在 25 MB/s 左右，优先检查热点、无线协商、信道和设备限制。

## 8. Python 异步概念的当前学习结论

面向用户解释时应保持以下准确边界：

- Python 确实支持真正的操作系统线程，`threading.Thread` 与 C 的线程概念相近。
- `asyncio` 是另一种并发方案，通常由一个操作系统线程运行事件循环。
- `async def` 定义协程函数。
- 调用协程函数产生协程对象。
- `asyncio.create_task(...)` 把协程注册为事件循环管理的任务。
- `await` 不会忽略当前语句，也不会让当前函数直接执行下一行。
- `await` 等待未完成 I/O 时，保存当前协程的执行位置和局部变量，把控制权交还事件循环。
- 事件循环可以用同一个线程运行其他已就绪任务。
- I/O 事件到达后，原任务从 `await` 后面恢复。
- 这可以类比为 Python 帮应用层封装了非阻塞 I/O、状态机和事件调度。
- 普通 CPU 计算、同步文件写入和 `time.sleep()` 仍会阻塞事件循环线程。

不要把以下概念混为一谈：

```text
协程函数：使用 async def 定义的函数
协程对象：调用协程函数产生的可暂停执行对象
Task：事件循环正式管理的协程
事件：网络可读、定时器到期等外部条件
事件循环：调度就绪任务并等待操作系统事件的管理器
并发：多个任务交错推进
并行：多个 CPU 执行单元同时工作
```

## 10. 建议下一步

按以下顺序推进，不要直接进行大规模重写：

1. 使用 `iperf3` 建立纯网络吞吐基线。
2. 在浏览器和服务端加入分阶段耗时统计：
   - 文件读取时间；
   - 分块网络发送时间；
   - 服务端请求体读取时间；
   - SHA-256 时间；
   - 临时文件写入时间；
   - `fsync` 时间。
3. 分别实测 8、16、32 MiB 分块，比较吞吐、内存和重传成本。
4. 将同步写盘和哈希从事件循环线程移出，或改成受控的流式处理。
5. 再评估是否需要多分块流水线或有限并行。
6. 如果改成并行分块，必须同步重新设计：
   - 服务端随机偏移写入；
   - 已完成分块位图；
   - 重复分块的幂等处理；
   - 完整性校验；
   - 最终提交条件；
   - 并发和内存上限。

性能优化必须以测量结果为依据，并保持现有的路径安全、权限检查、断点续传和原子提交语义。
